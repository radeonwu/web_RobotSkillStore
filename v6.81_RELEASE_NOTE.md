## What Changed in v6.81

### 1. RSS Skill Manifest v2.1 — Distribution Architecture

The skill download bundle format has been redesigned from a self-contained deployment recipe into a proper **two-block manifest** that separates frozen definition from live status, and delegates artifact distribution to existing package ecosystems.

#### Two-block structure

| Block | Content | Update cadence |
|---|---|---|
| `skill_definition` | Identity, orchestration interface, execution, hardware requirements, safety, robot profiles, composition graph, distribution pointer | Frozen at publish time |
| `skill_status` | Governance state, 30-day run stats, retrieved_at timestamp | Live — updated continuously |

#### Distribution pointer (`distribution` field in `skill_definition`)

The manifest no longer bundles artifact bytes. Instead it carries an open pointer to wherever the artifact lives in an existing registry:

| `implementation_type` | `distribution.type` | Backend |
|---|---|---|
| `algorithm` | `pypi` | PyPI / private index |
| `state_machine` | `inline` | No artifact — manifest IS the skill |
| `dnn_model` | `huggingface` | HuggingFace Hub |
| `hybrid` | `oci` | Docker Hub / GHCR |

#### Composition graph in manifest

`skill_definition.composition` now included in every downloaded manifest:
```json
"composition": {
  "can_follow":    ["pick_with_vision", "pick"],
  "enables":       ["torque_verify"],
  "parallel_safe": false,
  "idempotent":    false
}
```

Downloaded files are now named `{skill_id}-manifest-v2.1.json` (previously `-manifest-v2.json`).

### 2. Two New Public Platform API Endpoints

Both endpoints require no authentication.

#### Hardware Compatibility Oracle

```
GET /public/v1/skills/{skill_id}/compatibility
GET /public/v1/skills/{skill_id}/compatibility?robot=UR5e
```

- Without `?robot=`: returns full `compatible_robots` list
- With `?robot=UR5e`: returns `compatible: true/false` plus robot-specific parameter profile

#### Skill Composition Graph

```
GET /public/v1/skills/{skill_id}/composition
```

Returns `can_follow`, `enables`, `parallel_safe`, `idempotent` for orchestration planning.

### 3. Manifest Generation (Offline / Passkey Mode)

Client-side manifest generation in `en/skills.html` and `zh/skills.html` updated to produce v2.1 format, including:
- `skill_definition` / `skill_status` split
- `composition` stubs (populated from platform in JWT mode)
- `distribution` block derived from skill category

## Design Rationale

v6.81 reflects the decision that RSS Platform's unique value is:
1. **Hardware compatibility oracle** — no existing package manager answers "does this skill work on my robot?"
2. **Governance evidence registry** — lifecycle gates requiring real hardware runs, not CI pipelines
3. **Skill composition graph** — valid sequencing constraints for orchestrators

Distribution of artifact bytes is delegated to PyPI / HuggingFace Hub / OCI registries that already exist and are better at it.

## What Changed in v6.80

### 1. User Authentication System

v6.80 introduces a three-role authentication layer connected to the RSS Platform backend.

| Role | Access | How to obtain |
|------|--------|---------------|
| **Visitor** | Browse skill catalog, read docs | No login required |
| **End User** | Browse + download skill bundles | Register account (JWT) |
| **Developer** | All above + Platform & Developers pages | Register with invite code (JWT) |

**Auth modes:**

- **JWT mode** (recommended): `RSS_PLATFORM_BASE_URL` is set in `assets/config.js`. Registration and login call the platform `/auth/` API. Tokens are stored in `localStorage`.
- **Passkey mode** (offline / `file://`): No platform URL set. Access is granted by entering the correct passkey configured in `assets/config.js`.

### 2. Skill Download Gate

Downloading a skill bundle from the Skill Catalog page (`en/skills.html`, `zh/skills.html`) now requires End User or Developer login. Visitors see a sign-in prompt when clicking "Download bundle".

### 3. Sign-In / Register UI

- Account button in the top-right nav shows role badge when logged in.
- Register form includes role selection (End User / Developer).
- Developer registration requires an invite code.
- "Continue as Visitor" option always available.

## Configuration Reference

### Website — `assets/config.js`

```javascript
// Point to your running platform instance (no trailing slash)
window.RSS_PLATFORM_BASE_URL = "http://localhost:8080";

// Passkey mode fallback credentials (offline / file:// access)
window.RSS_USER_PASSKEY = "rss-user-2026";   // end-user passkey
window.RSS_DEV_PASSKEY  = "rss-dev-2026";    // developer passkey
```

### Platform — `docker-compose.yml` (api1 / api2 env)

| Variable | Default | Purpose |
|----------|---------|---------|
| `RSS_JWT_SECRET` | `rss-jwt-secret-change-in-prod` | HMAC key for signing JWT tokens — **change before production** |
| `RSS_JWT_EXPIRY_DAYS` | `7` | Token lifetime in days |
| `RSS_DEV_INVITE_CODE` | `rss-dev-invite-2026` | Code required to register a Developer account |

## Auth API Endpoints (Platform)

All endpoints are at `/auth/` and routed through the nginx gateway on port 8080.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/auth/register` | None | Register new account |
| `POST` | `/auth/login` | None | Login, returns JWT token |
| `GET`  | `/auth/me` | Bearer JWT | Get current user info |
| `GET`  | `/auth/admin/users` | X-API-Key (admin) | List all users |
| `PATCH`| `/auth/admin/users/{id}` | X-API-Key (admin) | Update user role / active |
| `DELETE`| `/auth/admin/users/{id}` | X-API-Key (admin) | Delete user |

**Register payload example:**
```json
{
  "name": "Alice",
  "email": "alice@example.com",
  "password": "securepassword",
  "role": "enduser"
}
```
For role `"developer"`, add `"invite_code": "rss-dev-invite-2026"`.

## Connecting Website to Platform

1. Start the platform:
   ```bash
   cd /path/to/RSS_Platform_v1.0.10
   docker compose up -d
   ```

2. Confirm the platform is running:
   ```bash
   curl http://localhost:8080/health
   # → {"status":"ok"}
   ```

3. Set the platform URL in the website config:
   ```javascript
   // assets/config.js
   window.RSS_PLATFORM_BASE_URL = "http://localhost:8080";
   ```

4. Open the website. Skills load live from the platform. Auth features activate automatically.

## Known Constraints

- The website must be served over HTTP (not `file://`) for the platform API fetch to work reliably. Use `python3 -m http.server 8000` or place behind the nginx gateway.
- `RSS_JWT_SECRET` and `RSS_DEV_INVITE_CODE` must be identical across `api1` and `api2` containers.
- Change `RSS_JWT_SECRET` before any production deployment.
- When connecting to a local platform via Cloudflare Tunnel, update `RSS_PLATFORM_BASE_URL` in `assets/config.js` with the tunnel URL. The default empty string causes the site to use the bundled static catalog.

Robot Skill Store (RSS) WhitepaperRSS Whitepaper (HTML)

要点
你将获得：把机器人技能当作可部署系统资产来管理的“平台视角”——版本绑定、可评测、可治理。
  • 平台优先：技能是可部署系统资产，而不是演示代码。
  • 版本化执行：每次 Run 都绑定不可变的 Skill Version。
  • 客观评测：以 Benchmark 判定成功，而非主观 Demo。
  • 运行时治理：策略门禁强制安全/授权/审批。
  • 证据与可审计：每次 Run 生成可追溯制品与记录。
推荐阅读顺序（RSS）
  1. 平台摘要：RSS 提供哪些能力（注册、运行、评测、治理）
  2. 架构摘要：平台为中心、协议强制
  3. 安全摘要：策略门禁与生产级护栏
概述
平台摘要
  • 注册表：技能、版本、制品、Profile。
  • Run：原子执行请求 + 不可变证据。
  • 评测：由 Benchmark 判定成功。
  • 治理：运行时策略门禁（安全/授权/审批）。
架构摘要
  • 平台为中心：平台拥有执行真相。
  • 协议强制：RSP 不变量必须执行。
  • 证据绑定：版本 + Profile + 结果。
安全摘要
  • 安全是系统强制，而非“最佳努力流程”。
  • 每个 Run 都经过不可绕过的门禁。
  • 审计链路可追溯、可复盘。
技能摘要
  • Skill 是契约化能力，不是“演示代码”。
  • 每次 Run 绑定不可变版本。
  • 公开页只展示聚合状态。
智能体摘要
  • 智能体提出 Run；平台强制策略。
  • 不能绕过语义与证据。

Robot Skill Protocol (RSP) WhitepaperRSP Whitepaper (HTML)

要点
RSP 的意义：为 RSS 提供不可变的不变量，让执行真相、评测与治理不依赖 UI、操作员或环境漂移。
  • 规范对象：Skill / Run / Profile / Policy / Evidence 的标准对象与语义。
  • 版本化契约：Skill 规格不可变，可跨时间可比较。
  • 运行期强制：所有动作都必须变成平台管理的 Run；Policy 决定允许发生什么。
  • 证据不可变：每次 Run 都产出可追溯制品（日志、指标、轨迹、结果）。
  • 可审计：评测与治理以证据为准，而非“演示口径”。
推荐阅读顺序（RSP）
  1. 概念：Skill / Agent / Runtime / Evidence
  2. Skill 规范：字段与语义
  3. 执行：绑定、强制、结果、证据
  4. 安全：技能级立场与机制
  5. Demo 映射：UR5 参考执行模型
概述
RSP 概览

RSP 是 RSS 内部被强制执行的协议层。它标准化对象与不变量，确保执行真相、评测与治理不依赖 UI、操作员或环境漂移。

  • 为什么：技能若只是 demo，很难规模化复用与治理。
  • 是什么：Skill / Run / Profile / Policy / Evidence 等标准对象与版本化语义。
  • 怎么做：运行时绑定规范到具体系统，并记录不可变结果与证据。
  • 带来什么：可对比、可审计、可跨仿真/真机复用。
概念

RSP 标准化的最小抽象（让不同团队/后端“说同一种语言”）：

  • Skill：能力契约（意图、约束、输入、可观测结果）——不是代码。
  • Run：一次执行实例，记录不可变并关联证据。
  • Profile：机器人/TCP/环境绑定与限制（确保“可比性”）。
  • Policy：运行时门禁与约束（决定允许发生什么）。
  • Evidence：日志/指标/轨迹/视频/溯源，使结果可审计。
  • Agent：基于规范与证据做决策（但不能绕过策略）。
Skill 规范

Skill Spec 定义字段与语义，使技能可部署、可评测、可治理且跨环境一致。

  • 标识：名称、版本、谱系、兼容性。
  • 意图/范围：要达成什么，以及明确禁止做什么。
  • 约束：几何、时序、力/扭矩、安全包络。
  • 输入/输出：声明参数与可观测结果（用于评测）。
  • 依赖：传感器/工具/环境假设等要求。
在 RSS 中，每个 Run 都绑定不可变的 Skill Spec 版本。
Schema

Schema 让规范可机器校验与自动化：

  • 校验：执行前拒绝不完整/不合法的记录。
  • 互操作：不同仿真/真机可消费同一契约。
  • 工具链：编辑器/校验器/CI/差异对比/升级。
  • 可追溯：稳定字段支持分析与证据检索。
执行

执行把抽象规范绑定到具体系统，并产出证据：

  • 绑定：选择 Skill Version + Profile + 参数 → 生成 Run 请求。
  • 强制：策略与约束在执行前/执行中检查。
  • 结果：由 Benchmark 度量，而非人为宣判。
  • 证据：制品归档并与 Run 关联，便于审计与复用。
安全

安全被定义为可执行、可测试的机制：

  • 技能级立场：声明风险、限制与必需护栏。
  • 策略门禁：审批、授权、环境就绪、合规。
  • 运行时约束：速度/力/空间限制、碰撞裕度、看门狗。
  • 事件处理：失败/重试/升级的标准化记录。
Demo 映射

Demo Mapping 把 UR5 参考流程映射到 RSP 对象：

  • 流程：pick → insert → tighten 作为组合工作流。
  • 版本：每个阶段独立版本化与评测。
  • Run：每次执行产出轨迹/视频/日志/指标等证据。
  • Policy：门禁阻止危险参数并强制约束。

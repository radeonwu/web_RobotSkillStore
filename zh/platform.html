RSS 平台能做什么

供应方注册技能，平台完成验证与认证形成信任，用户端消费证据完备的技能并部署到现场。

平台运行时流水线

以下流水线来自当前 RSS Platform 栈（v1.0.4.12）。验证会产出 Run + Evidence；认证/治理在不可变记录之上计算治理状态。

UI 入口（由 Gateway 暴露）：/viewer/（证据浏览），/public/v1/*（官网嵌入）。

平台核心组件

构建值得信赖的机器人技能的基础模块

Skill Registry（技能注册）

以稳定的能力契约发布技能（Schema、约束、版本）。在 API 服务中，Skill Spec 在入库前会做 Schema 校验。

  • 入库校验：JSON Schema（Draft 2020-12）通过 validate_skill_spec()
  • 创建/查询：POST /skills、GET /skills、GET /skills/{skill_id}
  • 治理分类：GET /skills/{skill_id}/{version}/taxonomy

验证引擎

每一次执行都成为平台管理的 Run，并绑定不可变的上下文与证据。

  • 创建 Run：POST /runs（或 POST /runs/submit）
  • 证据入库：POST /runs/{run_id}/artifacts
  • 事件/追踪：GET /runs/{run_id}/events
  • 读取真相：GET /runs/{run_id} 是唯一真相来源

Evidence & Artifact Store（证据与工件库）

工件可以存本地文件系统或 S3/MinIO（可配置）。每个工件都与 Run 绑定，从而让验证结果可审计、可复现。

  • 上传工件：POST /runs/{run_id}/artifacts
  • 列举工件：GET /runs/{run_id}/artifacts
  • 下载/链接：GET /artifacts/{artifact_id}
  • 预签名 URL：GET /artifacts/{artifact_id}/url

认证与治理

认证由证据窗口计算得到。平台聚合 Run outcome、约束违规、回归信号，并生成"治理快照"。

  • 技能治理快照：GET /skills/{skill_id}/{version}/governance
  • 策略评估（what-if）：POST /governance/policies/eval
  • 官网对外汇总：GET /public/v1/skills
  • 管理员刷新：POST /admin/public/v1/rollup

认证与公开读取：Anon-read / X-API-Key

RSS Platform 将官网可嵌入的公开读取与平台运维操作分离：前者用于展示与分享，后者用于执行、上传、管理刷新。

  • Public 读路径：/public/v1/* 设计为只读汇总（适合缓存与嵌入）
  • Anon-read 开关：当 RSP_AUTH_ALLOW_ANON_READ=1 时，网站可在不带凭据的情况下读取选定的 public 路由
  • X-API-Key：当 anon-read 关闭（或访问非 public 路由）时，客户端需携带 X-API-Key: <key>

环境变量与 Header

RSP_AUTH_ALLOW_ANON_READ=1
X-API-Key: <your-key>

经验法则：/public/v1 用于"可缓存的公开汇总"；所有会改变状态或访问原始证据的接口都要走认证。

工件存储后端切换

工件（视频、日志、遥测）可存本地或对象存储。平台通过 RSP_ARTIFACT_STORE 选择后端，并在 S3/MinIO 模式下通过预签名 URL让大文件直连传输。

  • 后端选择：RSP_ARTIFACT_STORE=local 或 RSP_ARTIFACT_STORE=s3
  • Local：工件写入平台卷；GET /artifacts/{artifact_id} 由平台直接回传
  • S3/MinIO：工件存为对象；平台下发带过期时间的 presigned URL

好处：Run 记录仍由平台统一治理，但大体量工件传输走对象存储直连，更适合团队规模化部署。

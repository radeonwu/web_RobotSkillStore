Reference Execution Model (Swimlane SVG)
Emphasis: explicit spec + runtime binding + constraint enforcement + auditable outcomes (sim/real consistent).

This demo is not a showcase. It is a reference execution model demonstrating how RSP governs skill execution across simulation and real robots — with reproducible bindings, constraint enforcement, and auditable outcomes.

Why a Reference Execution Model

Traditional robot demos optimize for visual results. Robot Skill Store demos optimize for correctness, reproducibility, and trust.

This UR5 example shows:

  • how a skill is specified (intent, constraints, outcomes)
  • how execution context is bound (robot base, TCP, environment)
  • how constraints are enforced during runtime
  • how outcomes are evaluated and recorded

System Setup

The reference setup includes:

  • a UR5 robot model
  • an explicit end-effector definition
  • a simulator or real robot runtime
  • a skill defined using RSP

The same skill definition is used across environments. Parameter changes are handled by runtime bindings, not by modifying the skill semantics.

Skill Specification

The UR5 demo skill explicitly defines:

  • task intent (e.g., pick, place, move)
  • motion and safety constraints
  • tool and kinematic assumptions
  • success and failure conditions

Nothing is implicit. Explicitness enables validation, comparison across runs, and safe reuse.

Runtime Parameter Binding

At execution time, the runtime binds:

  • robot base pose (e.g., base XYZ offset)
  • tool center point (TCP)
  • environment-specific parameters

These bindings can vary without changing the skill definition. This separation is critical for simulation-to-real transfer, reproducible experiments, and controlled deployment.

Execution Flow

  1. Skill is selected by an agent
  2. Runtime validates context and constraints
  3. Skill is executed on the UR5
  4. Outcomes are monitored and recorded
  5. Feedback is returned to the system

Each step is observable and auditable.

Simulation and Reality

The UR5 reference model treats simulation, testing, and real-world execution as part of the same semantic pipeline. A skill that fails in simulation should not be trusted in reality. This principle reduces risk and accelerates iteration.

Reuse and Extension

You are encouraged to reuse this reference model, adapt it to your own robots, and extend it with new skills. As long as the protocol is respected, the system remains interoperable.

Key takeaway

A robot demo becomes valuable only when it can be repeated, trusted, and extended.

What this demo is / is not

This demo is
  • Spec-first: driven by a skill spec & bindings
  • Fail-fast: constraint violations produce evidence
  • Auditable: evidence bundle for every run
  • Comparable: evidence supports regression comparisons
This demo is not
  • ROS / URScript tutorial
  • Robot controller implementation
  • Hardware bring-up guide
  • Application-specific workflow

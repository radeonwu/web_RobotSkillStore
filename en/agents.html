Agents

Agents decide what to try. RSS enforces what is allowed to actually happen.

In Robot Skill Store (RSS), agents are a decision layer for selecting and composing skills. They never bypass governance, evaluation, or execution truth—every action becomes a platform-managed Run with immutable evidence.

One non-negotiable rule

Agents may decide what to try. RSS enforces what is allowed to actually happen.

  • Agents produce candidate actions (which skill/version/profile/inputs).
  • RSS converts actions into Runs, applies policy gates, and records evidence.
  • Success is determined by benchmarks and outcomes—not by agent persuasion.

What an agent is (in RSS)

An agent is an orchestrator that reasons over goals, context, and historical evidence to choose which Skill Version to run, with which Profile and inputs.

Agents may / may not

Agents may

  • Select a Skill and Skill Version (within policy).
  • Bind Profile + parameters (robot / env / limits).
  • Compose multi-step workflows.
  • Branch / retry / escalate based on Run outcomes.
  • Compare candidates using benchmark evidence.

Agents may not

  • Execute code directly (no hidden paths).
  • Bypass policy gates (safety / license / approval).
  • Decide “success” (benchmarks do).
  • Edit records (Runs are immutable).
  • Redefine semantics (contracts are versioned).

Where agents sit in the system

  • Agent: selects & composes actions.
  • RSS Platform: validates, gates, schedules, and executes via runtime.
  • Runtime: binds specs to concrete robot/TCP/environment and enforces constraints.
  • Evidence: logs/metrics/trajectories/videos linked to the Run for audit and evaluation.

How RSS constrains agent actions

  • Policy gates: what is allowed (safety, approval, authorization).
  • Version binding: every Run binds to immutable Skill Version + Profile.
  • Objective evaluation: benchmark-defined success enables comparability.
  • Auditability: evidence is first-class; decisions are explainable and reviewable.

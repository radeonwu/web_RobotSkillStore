Canonical fields that agents and runtimes can trust

Why spec is not code

Implementations change; contracts must remain stable. RSP treats a skill spec as a capability contract that is readable by agents and enforceable by runtimes.

The 4 mandatory parts

  • Intent: what the skill is trying to achieve (goal-level semantics)
  • Context: assumptions about robot and environment
  • Constraints: boundaries that must be enforced during execution
  • Outcomes: observable success/failure, plus evidence requirements

Where it is enforced

The runtime is responsible for binding and enforcing. If a constraint cannot be enforced, the run must fail fast and produce evidence.

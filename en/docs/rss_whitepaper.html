Chapter 1 - Executive Summary

1.1 The State of Robotics Today

Robotics has entered an era of unprecedented visibility.

Research breakthroughs, open-source frameworks, simulation tools, and large-scale demonstrations have created the impression that robot capabilities are rapidly maturing.

Yet despite this progress, the deployment of robot skills in real production environments remains slow, fragile, and expensive.

Across industries, the same pattern repeats:

  • Demos succeed, deployments stall

  • Skills work in isolation, but fail at scale

  • Integration effort dominates innovation

  • Operational risk outweighs perceived benefit

The core issue is not a lack of algorithms or hardware.\ It is the absence of a system-level framework for managing robot skills as reliable, governable capabilities.

1.2 The Skill Problem Beneath the Surface

At the heart of the deployment challenge lies a deceptively simple question:

What is a robot skill?

In practice, robot skills are treated as:

  • Code fragments

  • ROS nodes

  • Learned policies

  • Demo pipelines

They lack:

  • A precise definition

  • Versioned semantics

  • Objective evaluation

  • Governance and accountability

As a result, robot skills cannot be trusted, compared, or deployed with confidence.

They remain engineering artifacts, not system assets.

1.3 Robot Skill Store: A Systemic Response

Robot Skill Store (RSS) is proposed as a platform-level solution to this problem.

RSS introduces a fundamental shift:

From skill-as-code to skill-as-system asset.

  • Evaluation is standardized and comparable

  • Governance is enforced by the platform

  • Trust emerges from accumulated evidence

RSS does not replace existing robotics frameworks.\ Instead, it provides the missing system layer that connects them into a deployable whole.

  • The RSS Website, which serves humans

The Platform ensures:

  • Execution truth

  • Governance and auditability

This separation is not cosmetic—it is essential for safety, scalability, and trust in physical systems.

1.5 Designed for Industrial Reality

RSS is designed with real industrial constraints in mind:

  • On-prem and air-gapped deployments

  • Long-lived production systems

  • Strict safety and compliance requirements

By maintaining consistent semantics across deployment environments, RSS enables innovation without sacrificing operational discipline.

Chapter 2 - The Skill Crisis in Robotics

2.1 Robotics Is Rich in Demos, Poor in Deployable Skills

Modern robotics appears vibrant:\ research papers, open-source repositories, demo videos, and impressive lab showcases are abundant.

Yet in practice, very few robot skills are actually deployable at scale.

Most “skills” in robotics today exist as:

  • Project-specific scripts

  • Tightly coupled ROS nodes

  • Ad-hoc policies trained for a single setup

  • Demo pipelines assembled for one-off demonstrations

They work once, in one environment, for one robot, under one set of assumptions.

When conditions change—even slightly—the skill breaks.

This gap between demonstration success and deployment readiness is the central crisis in robotics skills.

2.2 Skills Are Treated as Code, Not as Assets

In most robotics workflows, a skill is treated as code to be run, not as an asset to be managed.

As a result:

  • Skills have no explicit versioning semantics

  • Changes are rarely tracked or audited

  • Rollback is manual or impossible

  • Compatibility is undocumented

  • Trust is based on author reputation or anecdotal evidence

This is acceptable for research prototypes, but unacceptable for production systems.

Industries such as cloud computing, databases, and safety-critical software long ago learned that capabilities must be managed as assets, not as files.

Robotics, operating in the physical world, cannot afford to lag behind.

2.3 Demo-Centric Validation Breaks at Scale

Most robot skills today are validated visually:

  • The robot moves “roughly correctly”

  • The object is grasped “most of the time”

  • The alignment “looks good enough”

Such validation is inherently subjective.

It fails to answer critical questions:

  • Under what conditions does this skill fail?

  • How often does it succeed?

  • How does it compare to alternatives?

  • Has it regressed since the last change?

Without systematic evaluation, trust cannot be established, and without trust, deployment cannot scale.

2.4 Lack of Governance Prevents Industrial Adoption

In industrial environments, deploying a robot skill is not merely a technical act—it is an operational and legal decision.

Factories require answers to questions such as:

  • Who authored this skill?

  • Who approved its deployment?

  • What version is running on which machine?

  • What policies were checked before execution?

  • What evidence exists if something goes wrong?

Traditional robotics toolchains offer no coherent answers to these questions.

As a result, organizations either:

  • Freeze robot behavior and avoid updates

  • Rely on manual processes and tribal knowledge

  • Reject external skills entirely

None of these paths are sustainable.

2.5 Fragmentation Across Robots, Environments, and Vendors

Even when a skill is well-engineered, it is often:

  • Locked to a specific robot model

  • Tuned for a single scene

  • Dependent on proprietary interfaces

This fragmentation prevents reuse and ecosystem growth.

Without a common system that defines what a skill is, how it is evaluated, and how it is deployed, skills remain isolated artifacts rather than building blocks.

2.6 The Consequence: Skills Cannot Scale

Taken together, these issues lead to a single conclusion:

Robot skills, as they exist today, do not scale.

They do not scale across robots.\ They do not scale across environments.\ They do not scale across organizations.

Most importantly, they do not scale into production.

This is the problem Robot Skill Store is designed to address.

Chapter 3 - What Is a Robot Skill?

3.1 Why “Skill” Must Be Defined Precisely

In robotics, the term skill is widely used but rarely defined.

It may refer to:

  • A motion primitive

  • A ROS node

  • A script that controls actuators

  • A learned policy checkpoint

  • A demo pipeline assembled for a presentation

This ambiguity creates a serious problem:\ systems cannot be built on undefined concepts.

When “skill” means different things to different teams, it becomes impossible to:

  • Compare capabilities

  • Evaluate reliability

  • Govern deployment

  • Build reusable ecosystems

RSS starts by enforcing a precise definition of a robot skill—not as an implementation detail, but as a system contract.

3.2 Skill Is Not Code

A common misconception is to equate a robot skill with executable code.

Code is an implementation.\ A skill is a capability promise.

Code answers the question:

How is something done?

A skill must answer:

What capability is provided, under which conditions, and with what guarantees?

Multiple code implementations may exist for the same skill.\ Conversely, the same code may represent different skills under different constraints.

Therefore, in RSS, code is only one component of a skill—not its definition.

3.3 Skill Is Not a Demo

A demo demonstrates possibility.\ A skill must guarantee repeatability.

Demos are typically:

  • Manually curated

  • Environment-specific

  • Operator-assisted

  • Optimized for presentation

They answer:

Can this be shown once?

Skills must answer:

Can this be executed reliably across runs, environments, and operators?

RSS explicitly separates demonstration from skill definition.

A demo may showcase a skill.\ It does not define one.

3.4 Skill as a Contracted Capability

RSS defines a robot skill as a contracted capability.

A skill is not merely executable—it is describable, testable, and governable.

Formally, a robot skill in RSS is defined by:

  1. Declared Capability\ What the skill claims to achieve (e.g., “tighten a screw to torque X").

  2. Inputs\ Required parameters, references, or perceptions.

  3. Preconditions\ Assumptions about the environment, robot state, and tooling.

  4. Execution Boundary\ What the skill controls—and what it explicitly does not.

  5. Outputs\ Measurable outcomes, signals, or state changes.

  6. Failure Modes\ Known conditions under which the skill may fail.

  7. Evaluation Criteria\ Metrics and thresholds used to judge success.

This contract allows the skill to be reasoned about before it is executed.

3.5 Skill Versioning and Immutability

Once a skill is defined as a contract, immutability becomes mandatory.

A released skill version must not change its behavior silently.

Why?

  • Evaluation results must remain valid

  • Deployment decisions must be auditable

  • Rollback must be reliable

In RSS:

  • A Skill represents a logical capability

  • A Skill Version represents an immutable contract + implementation

Any change—no matter how small—creates a new version.

This principle aligns robotics with mature software and safety-critical systems.

3.6 Why This Definition Enables a Platform

Only with a precise definition of a robot skill can RSS enable:

  • Comparable evaluation

  • Meaningful benchmarks

  • Policy-based governance

  • Ecosystem reuse

Without a strict definition, RSS would collapse into:

  • A code repository

  • A demo gallery

  • Or a marketplace without trust

By defining skills as contracted, versioned, and evaluable capabilities, RSS establishes the foundation for everything that follows.

Chapter 4 - From Skill-as-Code to Skill-as-System Asset

4.1 The Limitations of Skill-as-Code

Treating robot skills as executable code has been the default approach for decades.

In this model:

  • A skill is a script, node, or package

  • Execution means “running the code”

  • Validation means “it worked during the demo”

  • Distribution means “share the repository”

This approach scales engineering effort, but it does not scale capability reliability.

The fundamental limitation is that code has no inherent semantics about capability.

Code alone does not specify:

  • What problem it solves

  • Under which conditions it is valid

  • How success or failure should be judged

  • Whether it is safe to deploy

As a result, every reuse attempt becomes a new integration project.

4.2 Why Reuse Fails Without System Context

In practice, most failed skill reuse attempts share a common pattern:

  • The code runs

  • The robot moves

  • But the outcome is unreliable or unsafe

This happens because skills depend on implicit system context:

  • Robot kinematics and limits

  • Tooling and calibration

  • Environment geometry

  • Perception assumptions

  • Timing and control characteristics

When these assumptions are undocumented and unenforced, reuse becomes guesswork.

RSS recognizes that context is not optional metadata—it is part of the skill itself.

4.3 Skill as a First-Class System Asset

RSS proposes a fundamental shift:

A robot skill is not code to be executed,\ but a system asset to be managed.

As a system asset, a skill must have:

  • Identity (what it is)

  • Version (what exactly it is)

  • State (where and how it is used)

  • History (how it performed over time)

  • Policy (under what rules it can run)

This mirrors how mature systems treat critical components:

  • Containers in cloud platforms

  • Images in manufacturing pipelines

  • Models in regulated ML systems

Robotics requires the same level of discipline.

4.4 What Changes When Skills Become Assets

This shift unlocks several critical capabilities:

1. Reproducibility

Every execution references a specific skill version and profile.

Results are no longer anecdotal—they are reproducible records.

2. Comparability

Different skills (or versions) can be benchmarked under the same conditions.

“Which one is better?” becomes a data-backed question.

3. Governability

Policies can be enforced at the asset level:

  • Who can run this skill?

  • On which robots?

  • In which environments?

4. Lifecycle Management

Skills can be:

  • Released

  • Deprecated

  • Rolled back

  • Certified

Systematically, not manually.

4.5 Why This Shift Requires a Platform

Managing skills as assets cannot be achieved through conventions alone.

It requires:

  • A canonical registry

  • Immutable versioning

  • Execution records

  • Evaluation infrastructure

  • Policy enforcement

In other words, it requires a platform.

Without a platform, “skill-as-asset” collapses back into:

  • Documentation conventions

  • Folder structures

  • Best-effort discipline

RSS exists precisely to provide this missing platform layer.

4.6 Setting the Stage for RSS Architecture

This chapter completes a critical transition:

  • Skills are no longer ephemeral demos

  • They are durable, governable system assets

With this shift established, the following questions naturally arise:

  • How are these assets modeled?

  • How are they executed and evaluated?

  • How are trust and policy enforced?

These questions lead directly to the RSS architecture and object model.

Chapter 5 - Robot Skill Store: System Overview

Robot Skill Store：系统整体概览

5.1 From Principles to a System

The previous chapters established three critical premises:

  1. Robot skills, as practiced today, do not scale

  2. A robot skill must be defined as a contracted, versioned capability

  3. Skills must be treated as first-class system assets

These premises imply a clear requirement:

A system is needed to manage the full lifecycle of robot skills.

Robot Skill Store (RSS) is that system.

RSS is not a single application, nor a collection of scripts.\ It is a coherent platform designed to manage robot skills from creation to deployment, across simulation and real-world environments.

Robot Skill Store（RSS）正是这样一个系统。

5.2 The Core Objective of RSS

The core objective of RSS is to make robot skills deployable with confidence.

This means:

  • Skills can be understood before execution

  • Skills can be evaluated objectively

  • Skills can be deployed under governance

  • Skills can evolve without losing trust

RSS does not aim to replace existing robotics frameworks.\ Instead, it provides the missing system layer that connects them.

5.3 What RSS Is — and What It Is Not

To avoid ambiguity, RSS explicitly defines its scope.

RSS is:

  • A skill-centric platform

  • A system-of-record for skill execution and evaluation

  • A governance layer for robot capabilities

  • A bridge between demos and production deployment

RSS is not:

  • A robot controller

  • A real-time motion planner

  • A replacement for ROS, MoveIt, or vendor SDKs

  • A mere website or marketplace

This distinction is critical to prevent architectural drift.

5.4 The Skill Lifecycle in RSS

RSS organizes robot skills around a clearly defined lifecycle:

  1. Definition\ A skill is defined as a contracted capability, with explicit assumptions and metrics.

  2. Packaging\ The implementation is packaged as an artifact bound to a specific skill version.

  3. Execution\ Runs are triggered in simulation or real environments via the Platform.

  4. Evaluation\ Results are measured, recorded, and compared using benchmarks.

  5. Governance\ Policies, permissions, and licenses determine where and how the skill may run.

  6. Deployment\ Approved skill versions are deployed into production environments.

Each stage produces system records that persist beyond individual executions.

5.5 Website and Platform as Complementary Roles

RSS realizes this lifecycle through two complementary system roles:

  • The Website, which serves humans

  • The Platform, which serves systems

The Website focuses on:

  • Discovery and understanding

  • Visualization and comparison

  • Human-triggered workflows

The Platform focuses on:

  • Execution and orchestration

  • Evaluation and record-keeping

  • Policy enforcement and auditability

Neither role is sufficient on its own.\ Together, they form a complete system.

5.6 Why RSS Is a Platform, Not a Product

A defining characteristic of RSS is that it is designed to be extended, not completed.

New skills, robots, environments, benchmarks, and policies can be added without redefining the system.

This openness allows RSS to function as:

  • A foundation for internal robotics teams

  • An integration layer for system integrators

  • A distribution and governance layer for skill providers

This is what distinguishes a platform from a product.

With the system overview established, the remaining questions become concrete:

  • What are the canonical objects managed by RSS?

  • How do these objects interact?

  • How does the system enforce execution truth and governance?

Chapter 6 - Website vs Platform: Separation of Concerns in Robot Skill Systems

6.1 Why Separation Matters in Robotics Systems

In traditional software products, the boundary between a website and a backend service is often a matter of implementation convenience.\ In robotics systems, however, this boundary becomes architecturally and operationally critical.

Traditional robotics workflows frequently blur this boundary:

  • Demos are executed directly from UI tools

  • Execution logic is embedded in scripts triggered by human operators

  • Results are validated visually rather than systematically

This approach may work for single demonstrations, but it fails fundamentally when robotics systems need to scale, be audited, or be deployed in production environments.

Robots operate in the physical world.\ Mistakes are not limited to software bugs; they translate into:

  • Physical damage

  • Safety risks

  • Production downtime

  • Liability exposure

For this reason, execution truth, evaluation, and governance must never depend on a human-facing interface.

6.2 The Role of the RSS Website

The RSS Website exists to serve human understanding and decision-making, not system execution.

Its primary responsibility is to answer questions such as:

  • What skills exist?

  • What capabilities do they claim?

  • How were they evaluated?

  • Under what conditions do they work?

The Website aggregates, visualizes, and contextualizes information produced by the Platform.\ It enables humans to explore the skill ecosystem, compare options, and initiate controlled actions.

Crucially, the Website does not own execution logic or system truth.

In RSS, the Website is allowed to:

  • Trigger executions

  • Visualize results

  • Guide onboarding workflows

But it is not allowed to:

  • Implement control loops

  • Decide execution success or failure

  • Store authoritative execution records

  • Enforce safety or policy gates

The Website is a control surface, not a control system.

6.3 The Role of the RSS Platform

The RSS Platform is the system-facing backbone of the Robot Skill Store.

It is responsible for all properties that must remain true regardless of who initiates an action, from where, or through which interface.

The Platform owns:

  • Skill lifecycle management

  • Execution orchestration

  • Evaluation and benchmarking

  • Policy enforcement and governance

  • Auditability and traceability

In other words, the Platform defines what actually happened.

If a skill is executed, the Platform records it.\ If a benchmark is passed or failed, the Platform determines it.\ If a policy gate blocks deployment, the Platform enforces it.

These guarantees must hold:

  • With or without a Website

  • In cloud, on-prem, or air-gapped environments

  • Under automation or human-triggered workflows

This is why the Platform must be API-first and interface-agnostic.

RSS 平台是 Robot Skill Store 的系统级骨干。

6.4 Why Execution Truth Must Live in the Platform

A common failure mode in robotics systems is UI-driven truth.

Examples include:

  • A demo button that implicitly defines success

  • A visual alignment that substitutes for quantitative metrics

  • A human operator manually deciding “this works”

These approaches are inherently non-reproducible and non-auditable.

RSS rejects UI-driven truth.

Instead, RSS enforces that:

  • Every execution is a Platform-managed Run

  • Every Run has inputs, outputs, logs, and metrics

  • Every decision can be traced to data and policy

This is the only way robot skills can evolve from demonstrations into deployable system assets.

6.5 Non-Negotiable Boundary Rules

To prevent architectural erosion, RSS enforces strict rules:

  1. No UI-only capabilities\ Any capability that affects execution, evaluation, or governance must exist as a Platform API.

  2. No duplicate truth\ The Website may cache or visualize data, but canonical records live in the Platform.

  3. No hidden execution paths\ If a skill can be executed, it must be traceable via Platform logs and identity.

  4. No policy bypass\ The Website cannot bypass safety, license, or governance checks under any circumstances.

These rules are not implementation preferences; they are system invariants.

6.6 Implications for System Design

This separation has concrete implications:

  • Platform services must be robust, testable, and deployable independently

  • Website development can iterate rapidly without risking system correctness

  • Third-party systems can integrate without depending on UI flows

Most importantly, this separation ensures that Robot Skill Store remains a platform, not merely a marketplace website.

Chapter 7 - Canonical Skill Model

7.1 Why a Canonical Model Is Necessary

A platform cannot function without a shared object model.

In robotics today, different teams describe the same concepts using incompatible abstractions:

  • A “skill” in one system is a script

  • In another, it is a ROS node

  • In a third, it is a trained policy

  • In a fourth, it is a demo pipeline

This fragmentation makes it impossible to:

  • Compare capabilities

  • Track changes over time

  • Enforce governance consistently

  • Build interoperable tooling

RSS introduces a canonical object model to establish a common language across systems, teams, and organizations.

7.2 Design Principles of the Skill Model

The canonical model in RSS follows four principles:

  1. Explicitness\ All assumptions must be made explicit.

  2. Immutability\ Released artifacts and versions must not change.

  3. Traceability\ Every execution must be traceable to identity, version, and configuration.

  4. Composability\ Objects should compose cleanly without hidden coupling.

These principles guide every object defined in the model.

7.3 Core Objects in RSS

RSS defines a small number of first-class objects.\ Each object has a precise responsibility and lifecycle.

7.3.1 Skill

A Skill represents a logical capability, independent of implementation.

It defines:

  • What the capability is

  • What outcome it claims to achieve

  • What class of tasks it belongs to

Examples:

  • tighten_screw

  • pickandplace

  • connector_insertion

A Skill does not specify:

  • How it is implemented

  • Which robot executes it

  • Which version is used

This separation allows multiple implementations to coexist under the same capability definition.

  • tighten_screw

  • pickandplace

  • connector_insertion

7.3.2 Skill Version

A Skill Version binds a capability definition to a concrete, immutable release.

It encapsulates:

  • A specific contract (inputs, assumptions, metrics)

  • One or more executable artifacts

  • Compatibility constraints

  • Evaluation results

Once released, a Skill Version must never change its behavior.

Any modification—code, parameters, or assumptions—creates a new version.

7.3.3 Artifact

An Artifact is a concrete executable package.

Examples include:

  • Container images

  • Python wheels

  • ROS2 packages

  • Vendor-specific binaries

Artifacts are always referenced through a Skill Version, never directly.

This prevents execution outside the governance boundary.

  • Python wheel

  • ROS2 包

7.3.4 Profile

A Profile captures runtime-specific configuration:

  • Robot model and kinematics

  • Tooling and calibration

  • Environment descriptors

  • Safety and limit parameters

Profiles allow the same Skill Version to run across different setups while preserving traceability.

7.3.5 Run

A Run represents a single execution of a skill.

It records:

  • Skill Version

  • Profile

  • Inputs

  • Outputs

  • Logs, metrics, and artifacts (e.g., video)

  • Status and timestamps

Runs are immutable historical records.

They form the empirical foundation of trust in RSS.

7.3.6 Benchmark

A Benchmark defines standardized evaluation procedures.

It specifies:

  • Test scenarios

  • Metrics

  • Thresholds

  • Pass/fail criteria

Benchmarks allow Skills and Skill Versions to be compared objectively.

7.3.7 Badge / Certification

A Badge represents a platform-issued trust signal.

Examples:

  • Verified

  • Production-Ready

  • Factory-Tested

Badges are derived from data, not claims.

7.4 Object Relationships

The canonical relationships are:

  • Skill → has many → Skill Versions

  • Skill Version → references → Artifacts

  • Skill Version → executed via → Runs

  • Runs → evaluated by → Benchmarks

  • Benchmarks → issue → Badges

This graph ensures that every claim can be traced to evidence.

  • Skill → 多个 → Skill Version

  • Skill Version → 引用 → Artifact

  • Skill Version → 产生 → Run

  • Run → 参与 → Benchmark

  • Benchmark → 产生 → Badge

7.5 Why This Model Enables Governance

Governance becomes feasible because:

  • Execution always references a version

  • Versions are immutable

  • Policies can be attached to objects

  • Audit trails are automatic

This shifts governance from manual process to system property.

7.6 From Object Model to Architecture

With a canonical object model, architecture choices become constrained and clear:

  • Which services own which objects

  • Which APIs expose which relationships

  • Where trust and policy are enforced

These questions are addressed in the following chapters on execution, evaluation, and governance.

Chapter 8 - Execution, Evaluation, and the Data Flywheel

8.1 Execution Is Not Just Running Code

In traditional robotics workflows, execution is often treated as a transient event:

  • A script is launched

  • The robot moves

  • The process ends

What remains is usually a video clip, a console log, or human memory.

RSS adopts a fundamentally different stance:

Execution is a first-class, persistent system event.

Every execution in RSS is a Run—a structured, immutable record that captures not only what was executed, but under what conditions and with what outcome.

This shift transforms execution from an ephemeral action into durable system data.

8.2 The Run as the Atomic Unit of Evidence

A Run is the atomic unit upon which trust is built in RSS.

Each Run captures:

  • Skill Version (what capability, exactly)

  • Profile (robot, environment, limits)

  • Inputs and parameters

  • Outputs and state changes

  • Logs, metrics, and artifacts (e.g., video)

  • Timestamps and identity

Runs are immutable.\ They are never edited, overwritten, or retroactively “fixed”.

If something changes, a new Run is created.

This immutability is essential for auditability and reproducibility.

8.3 From Runs to Evaluation

A single Run demonstrates behavior.\ Evaluation requires aggregation and comparison.

RSS separates execution from evaluation deliberately.

Evaluation is defined through Benchmarks, which specify:

  • Scenarios to test

  • Metrics to collect

  • Thresholds to enforce

  • Criteria for success or failure

Benchmarks operate over sets of Runs, not individual executions.

This allows RSS to answer questions such as:

  • How reliable is this skill across conditions?

  • How does this version compare to the previous one?

  • Which implementation performs better under constraint X?

8.4 Objective Metrics Replace Visual Judgment

In many robotics teams, validation still relies on visual inspection:

  • “It looks stable.”

  • “The motion seems smooth.”

  • “The grasp didn’t slip this time.”

Such judgments are inherently subjective and non-repeatable.

RSS replaces visual judgment with explicit, quantitative metrics.

Examples include:

  • Success rate

  • Completion time

  • Error tolerance

  • Force/torque limits

  • Positional deviation

These metrics turn vague impressions into measurable signals.

8.5 The Data Flywheel

As Runs accumulate and Benchmarks evaluate them, RSS creates a data flywheel:

  1. Execution generates data\ Each Run produces logs, metrics, and artifacts.

  2. Evaluation aggregates data\ Benchmarks transform raw runs into performance signals.

  3. Trust signals emerge\ Badges, rankings, and confidence scores are derived from evaluation.

  4. Better decisions are made\ Users choose skills based on evidence, not claims.

  5. Usage increases\ More execution generates more data, reinforcing the cycle.

This flywheel is self-reinforcing and improves over time.

8.6 Why the Flywheel Requires a Platform

The data flywheel cannot be sustained through ad-hoc tooling.

It requires:

  • Consistent data schemas

  • Immutable execution records

  • Centralized evaluation logic

  • Governance over data access

  • Long-term storage and lineage

Only a platform can provide these properties reliably.

Without a platform, data fragments across:

  • Logs on local machines

  • Videos in shared folders

  • Metrics in ad-hoc dashboards

The flywheel breaks.

  • Integrated into execution paths

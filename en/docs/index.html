Key Points
What you get: a platform view of robot skills as deployable system assets—versioned, evaluated, and governed.
  • Platform-first: skills are deployable system assets, not demo code.
  • Versioned execution: every run binds to an immutable skill version.
  • Objective evaluation: benchmarks define success, not subjective demos.

  • Evidence & auditability: every run produces traceable artifacts.
Summary
Platform summary

RSS is the product platform for deploying and operating skills at scale. It makes execution, evaluation, and governance platform-managed—not human-managed.

  • Registry: manage Skill / Version / Profile / Artifacts as first-class assets.
  • Runs: every action becomes a platform-managed Run with immutable records.
  • Evaluation: benchmark-defined success for comparability across time and teams.
  • Governance: approvals, authz, and safety gates enforced at runtime.
  • Evidence: logs/metrics/trajectories/video captured and linked to each Run.
Result: reliable rollout, measurable quality, and reusable capability—not “trust me” demos.
Architecture summary

RSS is platform-centered. RSP is enforced inside the platform so the same invariants hold across simulators and real robots.

  • Control plane: registry, versions, policies, and approvals.
  • Data plane: execution Runs + evidence capture + artifact storage.
  • Skill runtime: binds specs to robot/TCP/environment and enforces constraints.
  • Evaluation layer: benchmark definitions + scoring + dashboards.
  • Agent layer: planners/orchestrators propose Runs, but never bypass gates.
Rule: the platform is the source of truth; humans and UIs cannot “override reality.”
Safety summary

Safety is not a best-effort guideline. In RSS it is an enforceable mechanism.

  • Policy gates: approval/authz, environment readiness, compliance.
  • Runtime constraints: speed/force/workspace limits, collision margins, watchdogs.
  • Skill stance: hazards, limits, and required guards declared per skill.
  • Evidence: every safety-critical run yields traceable artifacts for audit.
Skills summary

Skills are deployable, versioned capability contracts. Code is implementation—contracts are what the platform governs.

  • Spec: intent, constraints, inputs, outcomes, requirements.
  • Versioning: immutable releases; comparison across time becomes meaningful.
  • Profiles: bind to robot/TCP/environment so runs are reproducible.
  • Artifacts: binaries/models/configs packaged and traceable.
  • Benchmarks: define success for evaluation and regressions.
Agents summary

Agents are a decision layer inside RSS. They select, compose, and retry skills based on goals and evidence—without bypassing execution truth.

  • May: choose candidates, bind profiles, compose workflows, branch/retry/escalate.
  • May not: execute code directly, bypass gates, edit records, redefine semantics.
  • Evidence-driven: compare candidates using benchmark outcomes, not persuasion.
Rule: Agents may decide what to try; RSS enforces what is allowed to actually happen.

Key Points
Why RSP exists: RSS needs non-negotiable invariants so execution truth and governance never depend on UI, operators, or environment drift.
  • Canonical objects: Skill / Run / Profile / Policy / Evidence with stable semantics.
  • Versioned contracts: skill specs are immutable and comparable across time.
  • Runtime enforcement: all actions become platform-managed Runs; policies gate what can happen.
  • Immutable evidence: every Run produces traceable artifacts (logs, metrics, trajectories, outcomes).
  • Auditability: evaluation and governance are evidence-driven, not “trust me” demos.
Summary
RSP overview

RSP is the enforced protocol layer inside RSS. It standardizes canonical objects and invariants so execution truth, evaluation, and governance never depend on UI, operators, or environment drift.

  • Why: robotics fails at scale when “skills” are demos without stable contracts and evidence.
  • What: canonical objects (Skill / Run / Profile / Policy / Evidence) with versioned semantics.
  • How: runtime binds specs to concrete systems and records immutable outcomes + artifacts.
  • Result: skills become comparable, auditable, and safe to reuse across simulators and real robots.
Concepts

Minimal abstractions RSP standardizes so different teams and backends speak the same language:

  • Skill: capability contract (intent, constraints, inputs, observable outcomes) — not just code.
  • Run: an execution instance with immutable records and links to evidence.
  • Profile: binding to robot/TCP/env + limits (what makes execution “the same”).
  • Policy: runtime gates and constraints (what is allowed to happen).
  • Evidence: logs/metrics/trajectories/video/provenance that make results auditable.
  • Agent: decision layer that reasons over specs and evidence (but cannot bypass policies).
Skill Spec

Skill Spec defines canonical fields and semantics so a skill can be deployed, evaluated, and governed consistently.

  • Identity: name, version, lineage, compatibility.
  • Intent & scope: what the skill tries to achieve and what it must never do.
  • Constraints: geometry, timing, force/torque, safety envelopes.
  • Inputs/outputs: declared parameters and observable outcomes (what will be measured).
  • Requirements: required sensors, tooling, environment assumptions.
In RSS, every Run binds to an immutable Skill Spec version.
Schema

Schema makes the spec machine-checkable and automatable.

  • Validation: reject malformed or incomplete skill/run records before execution.
  • Interoperability: different simulators/robots can consume the same contract.
  • Tooling: editors, linters, CI checks, diffing, and version upgrades.
  • Traceability: stable field names enable analytics and evidence search.
Execution

Execution binds abstract specs to concrete systems and produces evidence.

  • Binding: choose Skill Version + Profile + parameters → create a Run request.
  • Enforcement: policies and constraints are checked before/during execution.
  • Outcomes: results are measured by benchmarks, not by operator declaration.
  • Evidence: artifacts are stored and linked to the Run for audit and reuse.
Safety

Safety is defined as enforceable, testable mechanisms.

  • Skill-level stance: each skill declares hazards, limits, and required guards.
  • Policy gates: approval, authorization, environment readiness, compliance.
  • Runtime constraints: speed/force/workspace limits, collision margins, watchdogs.
  • Incident handling: standardized records for failures, retries, and escalations.

Two-minute mental model

  • RSS = the platform you deploy and operate.
  • RSP = the invariant layer RSS enforces (not a separate product choice).
  • Agents = a decision layer inside RSS Platform that proposes runs.
  • Runs = the atomic unit of evidence (immutable records + artifacts).
  • Policies = gates that cannot be bypassed at runtime.

Design stance

RSS stance
  • Production-first: built for repeatable deployment, not demos.
  • Governance-by-system: enforcement beats convention.
  • Evidence-driven: trust comes from runs and benchmarks.
RSP stance
  • Spec-first: define canonical objects and semantics.
  • Invariant enforcement: runtime gates are non-negotiable.
  • Auditability: outcomes are tied to versions, profiles, and evidence.
